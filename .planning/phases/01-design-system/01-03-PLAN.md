<task type="auto">
  <name>Task 1: Build Panel primitive component</name>
  <files>src/components/Panel.astro</files>
  <action>
    Create `src/components/Panel.astro`. This is an Astro component (not a utility class pattern) because the inner highlight ring requires a `::before` pseudo-element with `position: absolute` — Tailwind utility classes cannot express this.

    The component accepts an optional `class` prop that merges with the base `panel` class. Use `class:list` for merging.

    Key implementation requirements:
    - `.panel` MUST have `position: relative` (required for ::before absolute positioning)
    - `.panel` MUST NOT have `overflow: hidden` (this clips the inner highlight ring — see pitfall)
    - The `::before` inner highlight uses `rgba(255, 255, 255, 0.5)` — this creates the semi-transparent white ring visible against both the parchment surface and the linen surface
    - `box-shadow: 6px 6px 0px` — hard offset with zero blur (this is the GBA "printed shadow" look, not a soft drop-shadow)
    - Use `var(--color-shadow)` for box-shadow color, `var(--color-border)` for border, `var(--color-surface)` for background — all from the token system

    The component must use Astro's scoped `<style>` tag (not Tailwind utilities) for the panel styles so the ::before pseudo-element works correctly.

    After creating the file, verify the index page already has a `.panel` div (added in Plan 02 Task 2). If it does not exist, add it to `src/pages/index.astro` now. The Panel component itself does NOT need to be imported/used on the index page for the DS-06 test to pass — the `.panel` CSS class just needs to be defined and applied.

    However, also create an import of Panel in `src/pages/index.astro` and replace the raw `<div class="panel">` with `<Panel>` to validate the component works correctly:

<task type="auto">
  <name>Task 2: Build ThemeToggle client island and wire into Layout</name>
  <files>src/components/ThemeToggle.astro, src/layouts/Layout.astro</files>
  <action>
    Create `src/components/ThemeToggle.astro`. This is the ONLY component in Phase 1 that ships client-side JavaScript (Astro island with `client:load`).

    The component renders a `<button>` that:
    1. On click: reads current `data-theme` from `document.documentElement`, toggles fire↔leaf, writes to `localStorage('theme')`
    2. On load: syncs button label to current theme from localStorage (since FOUC script already applied the theme, just sync the label)

    Use text labels instead of emoji to avoid font rendering issues with DotGothic16:
    - Fire mode active: button shows "[ LEAF? ]" (offering to switch to Leaf)
    - Leaf mode active: button shows "[ FIRE? ]" (offering to switch to Fire)

    Or simpler — just show the current theme: "THEME: FIRE" / "THEME: LEAF"

    Style the button using design system tokens only (`var(--color-border)`, `var(--color-surface)`, `var(--color-text)`) — it should look like a small Panel-adjacent element.

    Add `aria-label` that updates dynamically: "Switch to Leaf theme" / "Switch to Fire theme".

    The script must be a regular `<script>` (NOT `is:inline`) — Astro will bundle it as a module, which is correct for a client island.

    After creating `ThemeToggle.astro`, update `src/layouts/Layout.astro`:
    1. Import ThemeToggle at the top of the frontmatter
    2. Add `<ThemeToggle client:load />` inside `<body>`, before `<slot />` (or in a `<header>` element)

    The `client:load` directive is what makes this a Astro island — it hydrates the component on page load.
  </action>
