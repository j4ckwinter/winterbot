## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | var(--color-world) — Fire: #8B3A2A / Leaf: #5A6B3A | Page background, world map texture |
| Secondary (30%) | var(--color-surface) — Fire: #F9EFE5 / Leaf: #EFF5E8 | Card (Panel) backgrounds, text on world |
| Accent (10%) | var(--color-accent-secondary) — Fire: #D4854A / Leaf: #8FA84A | Tag pills, dex numbers, "In Progress" label |
| Border / Muted | var(--color-border) — #66203D | Dividers, card borders |

**"In Progress" label color contract:**

- Background: var(--color-accent-secondary) with 0.25 opacity — `rgba` or CSS `color-mix` to distinguish from solid tech tags
- Text: var(--color-surface) — white-ish on amber/olive, passes WCAG AA
- Border: 1px solid var(--color-accent-secondary) — full opacity outline to frame the muted fill

**Rationale:** Solid var(--color-accent-secondary) is already used for technology tags. The "In Progress" badge must visually differ to avoid confusion — muted fill + outline achieves this without introducing new color tokens.

Accent reserved for: technology tag pill backgrounds, dex number text, "In Progress" label background tint, card-link--live text, card hover box-shadow

Destructive: not applicable — no destructive actions in this phase

## Component Anatomy

### "In Progress" Badge

Location in card: inside `.card-links` div, replacing the absent `liveUrl` link position. Rendered when `liveUrl` is undefined/null.

```
[ GitHub ]   [ IN PROGRESS ]
```

- Element: `<span>` (not `<a>` — not interactive, no href)
- Class: `card-badge--wip`
- Content: `IN PROGRESS`
- Font: DotGothic16 (matches tag + dex number — pixel-font aesthetic)
- Font size: 0.65rem (10.4px)
- Padding: 0.2rem 0.6rem (matches .tag)
- Background: color-mix(in srgb, var(--color-accent-secondary) 25%, transparent) OR rgba equivalent
- Color (text): var(--color-surface)
- Border: 1px solid var(--color-accent-secondary)
- Border-radius: 12px (matches .tag pill radius)
- Cursor: default (not pointer — non-interactive)
- Alignment: vertically centered with GitHub link via flexbox (existing .card-links flex row)

**Conditional rendering rule:** Badge renders if and only if `liveUrl` prop is absent/undefined. It never coexists with a Live link.

**Dual-theme:** No new tokens needed. var(--color-accent-secondary) already switches between Fire (#D4854A) and Leaf (#8FA84A) via the [data-theme="leaf"] override in global.css.

## Accessibility Contract

- `<span>` carries no implicit ARIA role — no `role="status"` or `aria-label` needed for a visual-only badge at this scope
- Screen readers will read "IN PROGRESS" as text content — acceptable
- Color alone does not convey the in-progress state: the text label is the primary signal (satisfies WCAG 1.4.1 Use of Color)
- Contrast: var(--color-surface) text on 25%-opacity var(--color-accent-secondary) over var(--color-surface) card background — effectively near-white text on a light amber tint. **Action required:** verify contrast ratio in implementation. If insufficient, use full-opacity background with var(--color-text) text instead.

|---------|------|

| Primary CTA | not applicable — no CTA in this phase |
| Empty state heading | not applicable — no empty state in this phase |
| Empty state body | not applicable |
| Error state | not applicable — static rendering, no async |
| Destructive confirmation | not applicable — no destructive actions |

**Copy rationale:** "IN PROGRESS" reads as a status descriptor consistent with the retro pixel-game theme. Alternatives considered and rejected:
- "Coming Soon" — vague, marketing-speak
- "WIP" — too informal, less clear to non-developers
- "No Live URL" — negative framing, exposes implementation detail
